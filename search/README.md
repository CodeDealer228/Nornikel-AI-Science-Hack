# Search module

Модуль `search` отвечает за построение поискового индекса и выполнение гибридного поиска по подготовленным chunks документов

> Перенесён из ветки `search-module-update` (коммит `e2a4183`) в `main`. При переносе:
> `build_index.py`/`build_index_qwen.py` получили аргументы `--chunks`/`--out-dir`
> (раньше пути были захардкожены константами `CHUNKS_PATH`/`OUT_DIR`); `get_source_document`
> в `build_index.py`/`build_index_qwen.py`/`searcher.py` теперь падает обратно на `chunk["doc_id"]`,
> если нет `chunk["provenance"]["source_document"]` — это нужно, чтобы модуль работал с
> chunks-форматом `llm_extraction/chunker.py` (`{doc_id, chunk_id, char_start, char_end, text}`,
> без вложенного `provenance`/`natasha`), а не только с форматом ветки `chunking`. Также
> удалены задвоенные определения `expand_lexical_query`/`get_synonym_matches` в `searcher.py`
> (мёртвый код от рефакторинга, вторая версия и так перекрывала первую) и добавлен CPU-fallback
> в `build_index_qwen.py` (было захардкожено `device="cuda"`). Проверено end-to-end на реальных
> чанках `parsed_data/texts/Статьи` (через `llm_extraction/chunker.py`) с настоящим
> `resources/synonyms.yaml` — BM25/RRF/synonym-expansion/numeric-extraction работают; dense-эмбеддинги
> проверены с фейковым embedder'ом (не отправлять ~1.1GB `intfloat/multilingual-e5-base` качать в CI) —
> перед реальным использованием нужно один раз прогнать `python -m search.build_index`, чтобы модель
> и веса скачались.

Основная идея пайплайна:

1. Документы заранее разбиваются на chunks
2. Для каждого chunk строятся dense embeddings
3. Параллельно строится BM25-индекс для лексического поиска
4. Пользовательский запрос нормализуется:
   - обрабатываются синонимы и доменные термины
   - англоязычный запрос переводится на русский (по возможности термины переводятся с помощью словаря)
5. Выполняется гибридный поиск:
   - dense search по эмбеддингам
   - BM25 search по токенам
   - объединение результатов через RRF (слияние через ранги)
6. Для найденных chunks дополнительно извлекаются числовые выражения (программно, работает неидеально, но быстро)

__init__.py

Служебный файл, который делает папку search Python-пакетом.

Благодаря этому можно запускать модули через:
python -m search.run_search
python -m search.build_index_qwen


build_index.py
Скрипт для построения базового поискового индекса (по умолчанию используется intfloat/multilingual-e5-base, быстрая модель)

порядок действий:
чтение подготовленных chunks;
извлечение текста из каждого chunk;
построение dense embeddings;
построение BM25-корпуса;
сохранение индекса на диск


build_index_qwen.py
Скрипт для построения индекса на базе модели Qwen/Qwen3-Embedding-0.6B (выбрана исходя из бенчмарка hugging face для multilingual эмбеддингов) 
Что делает:

Загружает chunks из подготовленных данных.
Извлекает текст chunk.
Кодирует chunks через Qwen embedding model.
Нормализует embeddings для cosine similarity.
Строит BM25-индекс по токенизированным chunk-текстам.
Сохраняет:
embeddings;
chunks;
BM25 index;
metadata индекса.

Запуск:
.\.venv\Scripts\python.exe -m search.build_index_qwen


run_search.py
CLI-обёртка для запуска поиска из командной строки
Принимает запросы на английском и русском языках (приоритетнее русский)
Если запрос на английском, то он переводится на русский 
(приоритетно перевод терминов с помощью словаря + остальное через Helsinki-NLP/opus-mt-en-ru, очень легкая локальная модель, можно взять что-то тяжелее)

Основные аргументы:
query — поисковый запрос.
--top-k — сколько результатов вывести.
--index-dir — путь к директории с индексом.
--full-text — печатать полный текст найденного chunk.
--show-query-plan — показать, как запрос был обработан перед поиском.

Пример: 
.\.venv\Scripts\python.exe -m search.run_search "очистка никелевых хлоридных растворов от железа меди свинца" --top-k 5 --index-dir parsed_data\search_index_qwen --full-text
.\.venv\Scripts\python.exe -m search.run_search "removal of iron copper lead from nickel chloride solutions" --top-k 5 --index-dir parsed_data\search_index_qwen --full-text --show-query-plan


searcher.py
Основной модуль поиска

В нём находится класс HybridSearcher, который объединяет:
dense search (поиск по эмбеддигам)
BM25 search (лексический поиск)
synonym expansion (в запрос для BM25 добавляются все эквивалентные записи имеющихся терминов для улучшения поиска)
query processing;
RRF fusion (объединение через ранги)
извлечение числовых выражений из найденных chunks (программный, работает неидеально)


query_processor.py
Модуль обработки пользовательского запроса перед поиском

Основные задачи:
Определить язык запроса.
Найти доменные термины через synonyms.yaml.
Для английских запросов построить русскоязычный search query.
Защитить технические термины от неправильного машинного перевода.
Построить две версии запроса:
dense_query;
bm25_query (bm25_query расширяется эквивалентными записями)

Для перевода используется glossary-aware подход:
Сначала ищутся известные термины из словаря.
Если весь короткий запрос покрыт словарём, переводчик не вызывается.
Если покрыта только часть запроса, переводятся только промежутки между терминами.
Найденные термины заменяются на русские canonical terms из словаря


synonyms.py
Модуль для работы со словарём синонимов (синонимы хранятся отдельно)

synonyms.py отвечает за:
загрузку YAML-файла;
хранение групп синонимов;
поиск совпадений в пользовательском запросе;
получение aliases по canonical_id;
удаление дублей;
подготовку терминов для query expansion.


mine_synonym_candidates.py

Скрипт для автоматического поиска кандидатов в синонимы.
Нужен для полуавтоматического расширения словаря.

Идея:

Пройтись по корпусу документов.
Найти часто встречающиеся термины, словосочетания, аббревиатуры.
Сгруппировать потенциально похожие выражения (определяется через скобки, слэш, тире между словами). 
Сохранить кандидатов для ручной/LLM проверки.

Этот скрипт не должен автоматически добавлять всё найденное в основной словарь без проверки,
потому что в техническом корпусе много шумных совпадений. 

Полученную информацию можно перевести в словарь например через LLM (для того чтобы отсеять плохих кандидатов)


numeric_extractor.py (неидеально работает)
Модуль для извлечения числовых выражений из найденных chunks

Он нужен, чтобы в выдаче сразу показывать важные параметры:
концентрации;
проценты;
температуры;
pH;
диапазоны;
массовые доли;
скорости;
давления;
единицы измерения

Пытается захватить контекст вокруг
Примеры: 
растворимость сульфата натрия: ~450 г/л
температура: 70 °C
и тд


requirements.txt
Использование:
.\.venv\Scripts\python.exe -m pip install -r search\requirements.txt



Ограничения:

1. Словарь синонимов основан только на той информации, что уже есть
2. Переводчик не является специализированным (сложные термины могут переводиться некорректно)
3. Extractor чисел неидеально работает
