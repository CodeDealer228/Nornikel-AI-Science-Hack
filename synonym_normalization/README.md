# synonym_normalization/ — Этап 2: Синонимы и дедупликация NER-графа

Постановка проблемы (ниже) написана после того, как проблема синонимов
неоднократно всплывала во время ручной разметки NER/RE (см.
`../ner_re_extraction/`). **Первая версия реализована**: `build_graph.py`
дедуплицирует сырые результаты `../llm_extraction/extraction_results_*.jsonl`
в файловый граф (`graph/nodes.jsonl` + `graph/edges.jsonl`) — см. раздел
"Реализация" ниже за деталями формата, гарантий и того, как это соотносится
с ещё не влитыми в `main` ветками `chunking` и `search-module-update`.

## Проблема

У каждой повторяющейся сущности в этом корпусе несколько поверхностных форм, и
разные статьи/авторы свободно выбирают разные варианты, часто в одном и том же
предложении:

| Каноническое понятие | Поверхностные формы, реально встреченные в корпусе |
|---|---|
| Печь Ванюкова конвертерная | `ПВК`, `печь Ванюкова конвертерная`, "конвертерная печь Ванюкова" |
| Печь Ванюкова обеднительная | `ПВО`, `печь Ванюкова обеднительная`, "обеднительная печь Ванюкова" |
| Цех электролиза никеля №2 | `ЦЭН-2`, "цех электролиза никеля №2" |
| Атомно-эмиссионная спектрометрия с индуктивно связанной плазмой | `ИСП-АЭС`, `ICP-AES` (пара RU/EN) |
| Электроэкстракция | `electrowinning` (пара RU/EN — прямо названная в самом задании хакатона) |
| Печь взвешенной плавки | `ПВП`, `fluidized bed furnace` (и русская аббревиатура, и английский перевод для одного и того же оборудования — тоже названы в задании) |
| Порошок никелевый трубчатых печей | `ПНТП` |
| Кислородно-воздушная смесь | `КВС` |
| АО «Кольская ГМК» | "Кольская ГМК", "Кольская гидрометаллургическая компания" |
| Институт Гипроникель | `ООО «Институт Гипроникель»`, "Гипроникель" |

Это не горстка особых случаев — это *обычное* состояние технического текста в
этой предметной области. Любой запрос вида "какие технические решения... описаны
в мировой практике" (форма запроса, которую задание требует поддерживать)
молча потеряет половину релевантной литературы, если `ПВК` и
`печь Ванюкова конвертерная` хранятся как два несвязанных узла.

## Почему это нужно решать на этапе извлечения, а не после

Если шаг NER сохраняет ту строку, которую он увидел, как идентичность узла, граф
получает N почти-дублирующих узлов на одну реальную сущность, каждый с
*частичным* набором отношений/фактов, которые на самом деле принадлежат одному
понятию. Объединение их постфактум требует заново выводить, какие поверхностные
формы кореферентны — строго сложнее, чем сделать это правильно один раз при
извлечении, когда аббревиатура и её расшифровка обычно находятся в одном и том же
абзаце (часто буквально "ПВК (печь Ванюкова конвертерная)").

## Предлагаемое решение

1. **Один канонический узел на понятие, со списком `aliases`**, а не отдельный
   узел на каждую поверхностную форму:
   ```
   (:Material {name: "никель", aliases: ["Ni", "nickel"]})
   (:Equipment {name: "печь Ванюкова конвертерная", aliases: ["ПВК"]})
   (:Process   {name: "электроэкстракция", aliases: ["electrowinning"]})
   ```
   Это ровно тот же формат, что уже используется в
   `../ner_re_extraction/ner_re_examples.md` (`главное название (уточнение)`) —
   ручная разметка уже выбирала каноническую форму и указывала алиас в скобках,
   это напрямую ложится на такую схему.

2. **Лексикон/газеттир, строящийся снизу вверх из уже извлечённого**, а не
   фиксированный словарь, написанный заранее. Каждый раз, когда извлечение видит
   паттерн вроде `X (Y)` или `X, также известный как Y` для распознанного типа
   сущности, `Y` добавляется в список алиасов `X` (или, если `Y` уже существует
   как отдельный канонический узел, оба узла объединяются, и граф сохраняет
   запись об этом объединении).

3. **Список алиасов нужен двум потребителям по разным причинам**, что
   соответствует половине парадигмы проекта про "semantic+lexic индекс":
   - **Lexical индекс** (точное/нечёткое совпадение строк, например
     Elasticsearch) — индексирует каждый алиас, а не только каноническое имя, так
     что поиск по `ПВК` и поиск по `печь Ванюкова конвертерная` оба попадают в
     один и тот же узел.
   - **Semantic индекс** (поиск по эмбеддингам) — каноническое имя + алиасы
     эмбеддятся вместе (либо алиасы используются, чтобы притянуть запрос в ту же
     область эмбеддингов, что и канонический термин), так что RU/EN парафразы, не
     имеющие общего корня (`электроэкстракция` / `electrowinning`), всё равно
     сходятся в один узел через смысл, а не через совпадение строк.

4. **Отдельно про пары RU/EN**: поскольку в этом корпусе смешаны русские и
   английские документы, список алиасов должен нести тег `lang` на каждый алиас
   (`{"text": "electrowinning", "lang": "en"}`), а не быть плоским списком строк —
   именно это позволяет сравнительному запросу "отечественная практика" vs
   "мировая практика" фильтровать корректно, не давая языку поверхностной формы
   протечь в фильтр по географии (русская статья может цитировать зарубежный
   процесс `electrowinning`, а англоязычный источник — описывать отечественный).

## Открытый вопрос — как он решён в `build_graph.py`

Должно ли объединение алиасов быть **полностью автоматическим** или требовать
**подтверждения человеком**? Требование задания "ручная корректировка графа
экспертами" предполагает, что второй вариант должен быть доступен как минимум.
Решение, реализованное сейчас:

* **Автоматически объединяются** только (а) точные совпадения нормализованной
  строки `type + canonical_name` — это не эвристика, а буквально одна и та же
  сущность под одним и тем же именем в разных чанках; (б) совпадения через
  `../resources/synonyms.yaml` — курируемый словарь, каждая запись которого уже
  когда-то прошла проверку человеком (промотирована из
  `synonym_candidates*.yaml`, см. ниже).
* **Всё остальное — только предложение**, не объединяется молча: паттерн
  аббревиатуры (`ПВК` рядом с `печь Ванюкова конвертерная` по инициалам) и
  нечёткое совпадение имён (`difflib`) пишутся в
  `../resources/synonym_candidates_from_ner.yaml` для ручного просмотра и
  промоушена в `synonyms.yaml`, если подтвердятся.

## Реализация

### Откуда взялись `resources/synonyms.yaml` и `resources/synonym_candidates.yaml`

Они **не написаны с нуля в этой сессии** — они перенесены (`git show
origin/search-module-update:resources/...`) из ветки `search-module-update`,
которая на момент переноса не была влита в `main` этого репозитория, но уже
содержала:

* `resources/synonyms.yaml` — 120 курируемых групп `canonical_id / type /
  canonical_name / aliases`, собранных вручную поверх корпуса;
* `resources/synonym_candidates.yaml` — 7451-строчный файл кандидатов,
  намайненных регэксп-паттернами (`full (ABBR)`, `ABBR — full`, `left / right`)
  прямо по сырому тексту чанков, ожидающих ручного промоушена в `synonyms.yaml`
  (см. `search/mine_synonym_candidates.py` в той же ветке).

Два случая из таблицы выше (`ЦЭН-2`, `КВС`) там отсутствовали и были добавлены
вручную в `synonyms.yaml` при переносе. Не переносился (пока) сам модуль
`search/` (гибридный поиск, BM25 + эмбеддинги) — только его словарь синонимов,
поскольку именно он нужен для дедупликации графа.

Ветка `chunking` тоже содержит собственный `synonym_normalization/` модуль
(`SynonymDictionary`, `normalize_entities`/`normalize_relations`,
`units_normalization.py`) — но он рассчитан на другую архитектуру: живая
нормализация *во время* извлечения (Natasha + LLM ensemble → `scripts/
ingest.py` → Neo4j), а не пост-обработка уже готового
`extraction_results_*.jsonl`, который есть в `main` сейчас. Из неё взята идея
нормализации единиц измерения (`units_normalization.py` — таблица `г/дм3` →
`г/л` и т.п.) и `EntityType`/`RelationType` enum (`ontology/ontology_types.py`)
как ориентир для того, какие типы сущностей вообще ожидаются — но сам код
модуля не скопирован, чтобы не тащить зависимость на pydantic-модели, которых
в текущем пайплайне `llm_extraction/` нет.

### `build_graph.py` — как это работает

```bash
cd synonym_normalization
python build_graph.py                      # по умолчанию: все ../llm_extraction/extraction_results_*.jsonl
python build_graph.py --input a.jsonl b.jsonl
```

Для каждой сущности из каждого входного файла:

1. Ищем `canonical_name` в `resources/synonyms.yaml` (нормализованное точное
   совпадение, O(1) по хэш-таблице). Если нашли — id узла = курируемый
   `canonical_id`, тип и каноническое имя берутся из словаря.
2. Если нет — id узла = `f"{type}::{normalize(canonical_name)}"`. Одинаковые
   по типу и нормализованному имени сущности из *любого* количества входных
   файлов схлопываются в один узел просто потому, что у них одинаковый ключ —
   без Union-Find и без прохода по всем парам.
3. `mentions` каждой сущности (сырые формы, которые видела LLM в конкретном
   чанке) добавляются в `aliases` узла с тегом `lang` (`ru`/`en`/`other`).

Почему не Union-Find по всем сущностям сразу (первая версия этого скрипта
делала именно так): при ~13k сырых сущностей (корпус "Статьи", один батч)
попарное сравнение через `difflib` на этапе поиска нечётких дублей не
завершилось за разумное время — точнее, оно долго молотило именно по типу
`Property` (числовые параметры, ~4000 сырых сущностей), где похожие по строке
имена почти никогда не являются реальными синонимами (значение уже точно
хранится в `attributes_history`, см. принцип 2 в корневом `CLAUDE.md`).
Решение: `Property` полностью исключён из нечёткого сопоставления
(`FUZZY_EXCLUDE_TYPES`), а точное совпадение узлов теперь не перебор, а прямое
хеширование по детерминированному ключу.

**Детерминированность** — id узла зависит только от его содержимого (курируемый
`canonical_id` либо `type::normalized_name`), никогда от порядка обработки или
счётчика. Повторный запуск на тех же входных файлах даёт побайтово идентичные
`nodes.jsonl`/`edges.jsonl` (проверено: одинаковый md5 на двух прогонах).
Добавление ещё одного `extraction_results_*.jsonl` в будущем (Обзоры, Доклады,
...) не переприсваивает существующие id — новые сущности либо схлопнутся в уже
существующий узел по тому же ключу, либо добавят новый; старые узлы не
меняются. Это и есть ответ на "таких jsonl будет много" — обработка растёт
линейно от общего числа сырых сущностей, а не квадратично, и не требует
пересчитывать всё с нуля каждый раз, чтобы остаться корректной.

### Формат вывода

`graph/nodes.jsonl` — один канонический узел на строку:
```json
{"id": "process_electrowinning", "type": "Process", "canonical_name": "электроэкстракция",
 "curated": true, "aliases": [{"text": "electrowinning", "lang": "en"}],
 "mention_count": 7, "source_chunks": [{"doc_id": "...", "chunk_id": "...", "local_id": "e4"}, ...],
 "attributes_history": [{"value": {"value_raw": "1,313", "operator": "=", "min": 1.313, "max": 1.313, "unit": null},
                          "source_chunk": {...}}]}
```
`curated` — узел разрешён через `resources/synonyms.yaml` (`true`) или получил
собственный `type::name`-ключ (`false`). `attributes_history` **не схлопывает**
числовые атрибуты разных упоминаний в одно значение — каждое версионируется
отдельно со своим `source_chunk` (принцип 8 в корневом `CLAUDE.md`: версионировать
факты, не перезаписывать).

`graph/edges.jsonl` — одна дедуплицированная тройка `(subject, predicate,
object)` на строку, с `occurrence_count` и списком `source_chunks`
(traceability, принцип 1).

`graph/merges.jsonl` — аудит: какие узлы реально объединили несколько разных
сырых поверхностных форм (`rule`: `curated_dictionary` или `exact_name`),
для проверки экспертом.

`resources/synonym_candidates_from_ner.yaml` — предложения объединений,
**не применённые автоматически** (см. решение открытого вопроса выше), в той
же схеме `candidates: [{aliases, source_type, count, examples}]`, что и
`resources/synonym_candidates.yaml` — чтобы куратор мог просматривать оба
файла одним и тем же процессом и продвигать подтверждённые пары в
`synonyms.yaml`.

### Файлы

| Файл | Роль |
|---|---|
| `canonicalizer.py` | `normalize()`/`lang_of()` — единая нормализация строк для всех модулей этого пакета |
| `synonym_dictionary.py` | Загружает `resources/synonyms.yaml`, отдаёт O(1) резолвинг `текст → SynonymGroup` |
| `build_graph.py` | Основной скрипт: сборка `graph/nodes.jsonl`/`edges.jsonl`/`merges.jsonl` + майнинг кандидатов |
| `graph/` | Выходной граф (см. формат выше) |
| `../resources/synonyms.yaml` | Курируемый словарь синонимов (перенесён из `search-module-update`, +2 записи) |
| `../resources/synonym_candidates.yaml` | Кандидаты, намайненные по сырому тексту (перенесено как есть, не используется `build_graph.py` напрямую — источник для будущего промоушена в `synonyms.yaml`) |
| `../resources/synonym_candidates_from_ner.yaml` | Кандидаты, намайненные `build_graph.py` по уже извлечённым сущностям |

### Известные ограничения / что дальше

* Резолвинг relations по `local_id` полагается на глобальную уникальность
  `chunk_id` в рамках корпуса (подтверждено эмпирически на `extraction_results_
  statyi_batch.jsonl` — 1334/1334 уникальных). Если формат `chunk_id` когда-то
  изменится и перестанет быть уникальным глобально, резолвинг тихо потеряет
  часть relations (сейчас неразрешённые просто логируются и пропускаются,
  принцип 7 — fail loud per-item, не молча).
* `resources/synonym_candidates.yaml` (намайненный по тексту) пока не
  скормлен обратно в `synonyms.yaml` — там 7451 кандидат, требующих ручного
  разбора; `build_graph.py` его не читает, только пишет отдельный файл со
  своими находками.
* Этот граф — файловый, для стадии до загрузки в графовую БД. Схема namespace
  узлов (`curated_id` или `type::name`) выбрана так, чтобы прямо ложиться на
  `neo4j_loader.py` из ветки `chunking` (там тоже узел уникален по `name` в
  рамках `label`), если/когда пайплайн перейдёт на Neo4j.
